SantéFlux — GDPR Privacy Pipeline

End-to-end privacy pipeline for a health-tech company — PII masking, salted hashing, frozen sensor removal, broadcast join, ghost user isolation, and an interactive City Stress Matrix for the board.

Platform · Databricks (Serverless)
Catalog · Unity Catalog
Language · PySpark
Domain · Healthcare / GDPR
Status · Complete
Date · March 2026

Overview

SantéFlux is a French health-tech company whose smart watch platform collects millions of bio-pings daily. When the Chief Privacy Officer found full names and raw user IDs in shared folders, analytics access was blocked entirely. The lead data engineer had 48 hours to build a privacy-compliant pipeline, remove corrupted sensor data, and deliver an interactive heart rate report to the board — or the Stress Spikes project would be shut down.

The pipeline had to satisfy two competing demands at once: strict GDPR compliance (mask names, salt and hash IDs, never expose raw PII) and a working analytical output (City Stress Matrix showing heart rate ranges by location and subscription tier).

Pipeline Architecture

Data Sources

Both files stored in a managed Unity Catalog volume at /Volumes/workspace/santeflux_case_study/raw_files/

CSV
vitals.csv

750,000 bio-ping records from smart watches. Contains User_ID, Full_Name, Timestamp, ville, Heart_Rate, and Status. Raw PII — requires masking and hashing before any shared use.

CSV
subs.csv

10,000-row subscription lookup — User_ID and Subscription_Type (Free or Premium). Small table, well suited for broadcast join to avoid shuffling the large vitals dataset.

STEP 01 Ingest Raw Files

Both CSVs ingested with inferSchema=True. Timestamp columns are correctly detected as TimestampType from the ISO format in the source — no explicit schema needed here.

from pyspark.sql.functions import *
from pyspark.sql.types import *

df_vitals = spark.read.csv(
    "/Volumes/workspace/santeflux_case_study/raw_files/vitals.csv",
    header=True, inferSchema=True
)

df_subs = spark.read.csv(
    "/Volumes/workspace/santeflux_case_study/raw_files/subs.csv",
    header=True, inferSchema=True
)
Why

Subscription data explored first with groupBy("Subscription_Type").count() to confirm the two tiers (Free / Premium) and check for unexpected values before building the join.

STEP 02 PII Masking — Names

Full names are partially masked at the point of ingestion. The first name is kept for readability; the last name is reduced to its first letter followed by asterisks. This satisfies the CPO's requirement while keeping the data usable for internal investigation.

df_masked = df_vitals.withColumn(
    "Full_Name",
    regexp_replace("Full_Name", r"^(\w+)\s(\w)(.*)", "$1 $2****")
)
# Pierre Dubois → Pierre D****
Why

Masking happens at the first transformation step — before any grouping, joining, or writing. Raw names never travel through the pipeline. The regex keeps first name + first initial intact so records remain traceable for debugging without exposing the full surname.

STEP 03 Salted Hashing — User IDs

Raw User_IDs cannot appear in any output. A fixed company salt is prepended before hashing — this stops rainbow table attacks where an attacker pre-computes hashes for all known IDs.

salt = "SanteFlux_Salt_2025"

# Hash vitals User_ID → store as new column hashed_name
df_hashed = df_masked.withColumn(
    "hashed_name",
    sha2(concat(col("User_ID"), lit(salt)), 256)
)

# Hash subs User_ID → same salt, same function, enables join
df_subs_upd = df_subs.withColumn(
    "hashed_name",
    sha2(concat(col("User_ID"), lit(salt)), 256)
).select("hashed_name", "Subscription_Type")
Why

Both tables must use the same salt and function — otherwise the hashes won't match and the join produces nulls. The salt must be shared with any team that needs to join back to this data. This is the trade-off: salting adds security but requires coordination across teams.

STEP 04 Frozen Sensor Detection

Some smart watch sensors were transmitting the same heart rate value for extended periods — a hardware fault, not real data. Including these rows would pull city averages toward a single stuck value. The detection approach: group by user and 10-minute time window, then flag windows where min Heart_Rate equals max Heart_Rate.

# Create 10-minute time windows
df_windowed = df_hashed.withColumn(
    "Time_Window", window(col("Timestamp"), "10 minutes")
).select("hashed_name", "Time_Window", "Heart_Rate", "ville")

# Find windows where sensor was stuck (all readings identical)
df_agg = df_windowed.groupBy("hashed_name", "Time_Window").agg(
    min("Heart_Rate").alias("min_rate"),
    max("Heart_Rate").alias("max_rate")
)
df_frozen = df_agg.filter(col("min_rate") == col("max_rate"))

# Remove frozen rows using left_anti join
vitals_clean_df = df_windowed.join(
    df_frozen,
    on=["hashed_name", "Time_Window"],
    how="left_anti"
)
Why left_anti

left_anti returns only rows from the left table that have no match on the right. It's the cleanest way to say "give me everything except what's in this exclusion list." A filter-based approach would require subqueries or a separate flag column — left_anti does it in one join.

STEP 05 Broadcast Join — Vitals + Subscriptions

The cleaned vitals table (750k rows) needs to be joined to the subscription lookup (10k rows). A standard join would shuffle both tables across the network. Broadcasting the small subscriptions table instead sends one copy to every executor — the large table never moves.

df_joined = vitals_clean_df.join(
    broadcast(df_subs_upd),
    vitals_clean_df.hashed_name == df_subs_upd.hashed_name,
    how="left"
).select(
    vitals_clean_df.hashed_name,
    "Time_Window",
    "Heart_Rate",
    "ville",
    "Subscription_Type"
)
Why broadcast

Confirmed in the physical plan — PhotonBroadcastHashJoin BuildRight with EXECUTOR_BROADCAST on the subs side. No Exchange node on the large vitals table. The subscription lookup at ~125 KB is well within safe broadcast range (default threshold: 10 MB).

STEP 06 Ghost User Isolation

Some bio-pings in the vitals file had no matching record in the subscriptions file — users who exist in the sensor data but never registered a subscription. These rows return null for Subscription_Type after the left join. Including them in the pivot would create a null column in the board report.

# Isolate ghost users — no subscription match
df_ghost = df_joined.filter(col("Subscription_Type").isNull())

# Clean data — matched users only, safe for board report
df_clean = df_joined.filter(col("Subscription_Type").isNotNull())
Why

Ghost users are isolated, not deleted. df_ghost is available for investigation — the data team can trace these IDs back to the source system to understand why they have no subscription record. The board report is built exclusively from df_clean.

STEP 07 City Stress Matrix — Pivot

The board needs to see heart rate ranges by city and subscription tier in a single table. Pivot turns the two subscription values (Free, Premium) into columns, with max and min Heart_Rate as the metrics.

df_final = df_clean.groupBy("ville").pivot("Subscription_Type").agg(
    max("Heart_Rate").alias("max_Heart_Rate"),
    min("Heart_Rate").alias("min_Heart_Rate")
)
display(df_final)
Why pivot creates shuffle

Pivot is a wide transformation — it needs all rows for the same city and subscription type to land on the same executor before it can compute the aggregation. This triggers a shuffle. Removing ghost users and frozen rows beforehand reduces the data volume entering that shuffle, which is why the order of steps matters.

STEP 08 Interactive Board Widget

The board needs to filter the report by city without re-running the full pipeline. A Databricks dropdown widget captures the selected city. Only the display cell re-runs when the selection changes — the full pipeline from ingestion to pivot has already executed and df_final is in memory.

# Create dropdown widget (runs once)
dbutils.widgets.dropdown("ville", "Paris", ["Paris", "Lyon", "Marseille"])

# Read widget value and display filtered result (re-runs on change)
selected_ville = dbutils.widgets.get("ville")
display(
    df_final.filter(col("ville") == selected_ville)
)
Why

The widget is the notebook equivalent of a slicer in Power BI — the underlying data is already computed, and the control just filters the display. Changing the dropdown triggers only the cell that reads the widget value, not the entire notebook.

Key Learnings

Salt Before You Hash

SHA-256 without a salt is vulnerable to rainbow table attacks — an attacker can pre-compute hashes for all known IDs. Concatenating a company secret before hashing stops this. The trade-off: every team joining on this data must use the same salt.

left_anti for Exclusion Lists

The cleanest way to remove rows that match an exclusion list. Returns only left-side rows with no match on the right. One join replaces what would otherwise be a subquery or a flag column followed by a filter.

Broadcast Join — Know the Limit

Sending the small table to every executor eliminates the shuffle entirely. Safe up to ~200 MB. The default auto-broadcast threshold is 10 MB — tune via spark.sql.autoBroadcastJoinThreshold. Confirmed working when the physical plan shows BroadcastHashJoin with no Exchange on the large table.

Ghost Users from Left Joins

A left join preserves all rows from the driving table regardless of match. Any row with a null in the joined columns is a ghost — exists in the source but not in the lookup. Always check for and isolate these before aggregating.

Pivot Creates Shuffle

Pivot is a wide transformation. It needs all rows for the same group key to land on the same executor before aggregating. The order of steps matters — remove bad data before the pivot, not after, to minimise the data crossing the network.

Read Physical Plans Bottom to Top

Physical plans execute from the bottom node up. FileScan at the bottom, result stage at the top. Exchange nodes indicate shuffles. PushedFilters on FileScan confirms predicate pushdown — the filter ran at read time, not after loading all rows into memory.
