Green Grid — Smart Billing Pipeline

End-to-end PySpark pipeline on Databricks — from raw file ingestion to a silver-layer billing table with peak-hour pricing logic.

Platform · Databricks (Serverless)
Catalog · Unity Catalog
Language · PySpark
Status · Complete
Date · March 2026

Overview

Green Grid is a fictional energy company with three raw data sources: customer PII records, smart meter readings, and pricing plans. The goal was to build a complete ingestion-to-silver pipeline that produces a per-reading bill amount with peak-hour pricing applied.

This case study covers every step a data engineer would face on a real project: schema enforcement, data quality quarantine, PII protection, multi-dataset joins, and business logic transformations — all written in PySpark on Databricks.

Pipeline Architecture

Raw Files (Unity Catalog Volume)
  customer_pii.csv  ·  meter_readings.json  ·  pricing_plans.csv
    ↓ Ingest
DataFrames — df_pii · df_readings · df_plans
    ↓ Data Quality
Quarantine_Readings (bad records: kwh < 0 or kwh ≥ 100)
clean_readings (valid records only)
    ↓ Join
df_joined — clean readings + PII + pricing plans
    ↓ PII Hashing
df_hashed — CustomerEmail replaced with SHA-256 hash
    ↓ Business Logic
Hour extracted · Peak rate applied · Bill_Amount calculated
    ↓ Write
Silver_SmartBill (Delta managed table)

Data Sources

All three files were stored in a managed Unity Catalog volume:

  • customer_pii.csv — Customer records including email addresses (PII)
  • meter_readings.json — Smart meter kWh readings with timestamps (multiLine JSON array)
  • pricing_plans.csv — Plan names and base price per kWh

Step 1 — Ingest

CSV files used inferSchema=True for type detection. The JSON meter readings required an explicit StructType schema to enforce TimestampType and DoubleType — JSON partial inference leaves timestamps as strings.

Explicit Schema — meter_readings.json

schema_readings = StructType([
    StructField("kwh",       DoubleType(),    True),
    StructField("meter_id",  StringType(),    True),
    StructField("status",    StringType(),    True),
    StructField("timestamp", TimestampType(), True)
])

df_readings = spark.read.json(
    "/Volumes/workspace/green_grid_case_study/raw_files/meter_readings.json",
    multiLine=True,
    schema=schema_readings
)

Why explicit schema? Without it, timestamp comes back as StringType — which breaks any time-based function downstream (hour(), date filtering, window functions). Explicit schema is the production pattern: one pass over the file, full type control, no surprises.

Step 2 — Data Quality Quarantine

Before any transformation, bad records were isolated into a quarantine table. This is a standard medallion architecture pattern — never transform dirty data, always preserve it for investigation. 

# Quarantine: readings outside valid range
df_quarantine = df_readings.filter(
    (df_readings.kwh < 0) | (df_readings.kwh >= 100)
)
df_quarantine.write.mode("overwrite").saveAsTable(
    "workspace.green_grid_case_study.Quarantine_Readings"
)

# Validate the write
df_test = spark.read.table("workspace.green_grid_case_study.Quarantine_Readings")
df_quarantine.count() == df_test.count()

The count assertion confirms no rows were dropped or duplicated during the write — a lightweight but important validation step before moving downstream.

Step 3 — Clean Readings

The inverse of the quarantine filter — only readings with valid kWh values pass through to the clean table. 

df_clean = df_readings.filter(
    df_readings.kwh.isNotNull() &
    (df_readings.kwh >= 0) &
    (df_readings.kwh < 100)
)
df_clean.write.mode("overwrite").saveAsTable(
    "workspace.green_grid_case_study.clean_readings"
)

Step 4 — Join Three Datasets

Two left joins chained in a single statement — clean readings to PII on meter_id, then PII to pricing plans on PlanID. Ambiguous columns dropped explicitly to avoid duplicates downstream. 

df_joined = df_clean \
    .join(df_pii,   df_clean.meter_id == df_pii.MeterID,    "left") \
    .join(df_plans, df_pii.PlanID     == df_plans.PlanID,   "left") \
    .drop(df_clean.meter_id) \
    .drop(df_pii.PlanID)

Step 5 — Hash PII

Customer email addresses are PII — they cannot appear in a silver table that downstream consumers may access. SHA-256 hashing preserves the ability to join back to the source system while removing the raw identifier. 

df_hashed = df_joined \
    .withColumn("CustomerHashed", sha2(df_joined.CustomerEmail, 256)) \
    .drop(df_joined.CustomerEmail)

Step 6 — Peak Pricing & Bill Calculation

Business rule: readings between 18:00 and 21:00 (peak hours) are charged at 2× the base rate. The hour() function extracts the hour from the timestamp column, and when().otherwise() applies the conditional rate — equivalent to SQL CASE WHEN. 

# Extract hour from timestamp
df_final1 = df_hashed.withColumn("Hour", hour(df_hashed.timestamp))

# Apply peak pricing
df_finalrate = df_final1.withColumn(
    "FinalRate",
    when(
        (df_final1.Hour >= 18) & (df_final1.Hour <= 21),
        df_final1.BasePricePerKwh * 2
    ).otherwise(df_final1.BasePricePerKwh)
)

# Calculate bill amount
df_billamount = df_finalrate.withColumn(
    "Bill_Amount",
    round(df_finalrate.kwh * df_finalrate.FinalRate, 2)
)

Step 7 — Write Silver Table

Final output written as a managed Delta table in Unity Catalog. Read-back confirms the write succeeded. 

df_billamount.write.mode("overwrite").saveAsTable(
    "workspace.green_grid_case_study.Silver_SmartBill"
)

df_silver_tbl = spark.read.table("workspace.green_grid_case_study.Silver_SmartBill")
display(df_silver_tbl)

Key Learnings

Explicit Schema vs inferSchema

inferSchema scans the file twice. Explicit StructType gives full type control in one pass — the production pattern for any file where types matter (timestamps, decimals).

multiLine JSON

Spark expects newline-delimited JSON by default. Standard JSON arrays (starting with [) require multiLine=True — without it you get a corrupt record error.

Quarantine Before Transform

Isolate bad records first, then process clean data. Preserving quarantined rows in a separate table means nothing is silently dropped — you can always go back and investigate.

PII in Pipelines

Raw PII identifiers should never reach a silver layer. SHA-256 hashing (sha2()) replaces the raw value while preserving join capability back to source systems.

Chained Joins

Two left joins chained in one statement. Ambiguous column names from multiple tables must be dropped explicitly using the DataFrame reference — not just the column name string.

when().otherwise()

PySpark's equivalent of SQL CASE WHEN. Conditions must use & / | (bitwise), not Python's and / or — which silently fail on Spark Column objects.

Tables Produced

  • clean_readings — Valid meter readings only
  • Silver_SmartBill — Final enriched table with hashed PII, plan details, and calculated bill amounts
