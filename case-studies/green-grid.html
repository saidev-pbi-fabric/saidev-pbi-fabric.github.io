Green Grid — Smart Billing Pipeline

End-to-end data pipeline — raw file ingestion through to a silver-layer billing table with peak-hour pricing, PII hashing, and data quality quarantine.

Overview

Green Grid is a fictional energy company with three raw data sources. The goal: build a complete ingestion-to-silver pipeline that produces a per-reading bill amount with peak-hour pricing applied — covering every step a data engineer would face on a real project.

Pipeline Architecture

Raw Files (Unity Catalog Volume)
  customer_pii.csv  ·  meter_readings.json  ·  pricing_plans.csv
    ↓ Ingest
DataFrames — df_pii · df_readings · df_plans
    ↓ Data Quality
Quarantine_Readings (bad records: kwh < 0 or kwh ≥ 100)
clean_readings (valid records only)
    ↓ Join
df_joined — clean readings + PII + pricing plans
    ↓ PII Hashing
df_hashed — CustomerEmail replaced with SHA-256 hash
    ↓ Business Logic
Hour extracted · Peak rate applied · Bill_Amount calculated
    ↓ Write
Silver_SmartBill (Delta managed table)

Data Sources

JSON

CSV

Plan names and base price per kWh. Joined to readings via PlanID from customer records.

STEP 01 Ingest Raw Files

CSV files used inferSchema=True. The JSON meter readings required an explicit StructType schema — JSON partial inference leaves timestamps as strings, which breaks all time-based functions downstream.

from pyspark.sql.functions import *
from pyspark.sql.types import *

# CSV — inferSchema handles type detection
df_pii   = spark.read.csv(".../customer_pii.csv",   header=True, inferSchema=True)
df_plans = spark.read.csv(".../pricing_plans.csv",   header=True, inferSchema=True)

# JSON — explicit schema required to enforce TimestampType
schema_readings = StructType([
    StructField("kwh",       DoubleType(),    True),
    StructField("meter_id",  StringType(),    True),
    StructField("status",    StringType(),    True),
    StructField("timestamp", TimestampType(), True)
])

df_readings = spark.read.json(
    "/Volumes/workspace/green_grid_case_study/raw_files/meter_readings.json",
    multiLine=True,
    schema=schema_readings
)
Why

Explicit schema = one file scan, full type control. inferSchema scans the file twice and still leaves timestamps as strings. For anything going to production, define the schema explicitly.

STEP 02 Quarantine Bad Records

Before any transformation, invalid readings are isolated into a separate table. Standard medallion pattern — never transform dirty data, always preserve it.

# Isolate bad records: negative or unrealistically high kWh
df_quarantine = df_readings.filter(
    (df_readings.kwh < 0) | (df_readings.kwh >= 100)
)
df_quarantine.write.mode("overwrite").saveAsTable(
    "workspace.green_grid_case_study.Quarantine_Readings"
)

# Validate: confirm row count matches what was written
df_test = spark.read.table("workspace.green_grid_case_study.Quarantine_Readings")
df_quarantine.count() == df_test.count()  # must return True
Why

The count assertion is a lightweight write validation — confirms no rows were silently dropped or duplicated. Visual spot-check via display() alone is not enough.

STEP 03 Filter Clean Readings

Inverse of the quarantine filter. Nulls handled explicitly — they pass through range checks silently if not caught.

df_clean = df_readings.filter(
    df_readings.kwh.isNotNull() &
    (df_readings.kwh >= 0) &
    (df_readings.kwh < 100)
)
df_clean.write.mode("overwrite").saveAsTable(
    "workspace.green_grid_case_study.clean_readings"
)
Why

NULL ≠ False in Spark. NULL < 0 evaluates to NULL, not True — so nulls pass through a range filter into the clean dataset unless you explicitly check isNotNull().

STEP 04 Join Three Datasets

Two left joins chained in one statement. Ambiguous column names dropped using DataFrame references — not string names, which would fail silently.

df_joined = (
    df_clean
    .join(df_pii,   df_clean.meter_id == df_pii.MeterID,  "left")
    .join(df_plans, df_pii.PlanID == df_plans.PlanID,     "left")
    .drop(df_clean.meter_id)   # drop ambiguous column by DataFrame ref
    .drop(df_pii.PlanID)       # drop ambiguous column by DataFrame ref
)
Why

After joining tables that share column names, dropping by DataFrame reference (df_clean.meter_id) is unambiguous. Dropping by string name ("meter_id") when duplicates exist throws an error.

STEP 05 Hash PII

Customer email is PII and cannot appear in a silver table that downstream consumers access. SHA-256 replaces the raw value while preserving join-back capability to the source system.

df_hashed = (
    df_joined
    .withColumn("CustomerHashed", sha2(df_joined.CustomerEmail, 256))
    .drop(df_joined.CustomerEmail)   # raw email removed — never reaches silver
)
Why

SHA-256 is one-way — the original email cannot be recovered from the hash. But the same email always produces the same hash, so you can still join back to source systems if needed.

STEP 06 Apply Peak Pricing & Calculate Bill

Business rule: readings between 18:00–21:00 are charged at 2× the base rate. hour() extracts the hour component. when().otherwise() is PySpark's equivalent of SQL CASE WHEN.

# Extract hour from TimestampType column
df_final1 = df_hashed.withColumn("Hour", hour(df_hashed.timestamp))

# Apply peak rate: 18:00–21:00 = 2x BasePricePerKwh
df_finalrate = df_final1.withColumn(
    "FinalRate",
    when(
        (df_final1.Hour >= 18) & (df_final1.Hour <= 21),
        df_final1.BasePricePerKwh * 2
    ).otherwise(df_final1.BasePricePerKwh)
)

# Bill amount = kWh consumed × final rate, rounded to 2 decimal places
df_billamount = df_finalrate.withColumn(
    "Bill_Amount",
    round(df_finalrate.kwh * df_finalrate.FinalRate, 2)
)
Why

Conditions in when() must use & / | (bitwise operators), not Python's and / or. Python's logical operators don't work on Spark Column objects and will produce wrong results silently.

STEP 07 Write Silver Table

Final output written as a managed Delta table in Unity Catalog. Read-back immediately after confirms the write completed successfully.

# Write to Unity Catalog as managed Delta table
df_billamount.write.mode("overwrite").saveAsTable(
    "workspace.green_grid_case_study.Silver_SmartBill"
)

# Read-back validation
df_silver_tbl = spark.read.table("workspace.green_grid_case_study.Silver_SmartBill")
display(df_silver_tbl)
Why

Read-back after write is a quick sanity check that the table exists, is queryable, and returns data in the expected shape — before any downstream process depends on it.

Key Learnings

Explicit Schema vs inferSchema

inferSchema scans the file twice and still misses timestamps. Explicit StructType = one pass, full type control — the production pattern.

multiLine JSON

Spark expects newline-delimited JSON by default. Standard arrays ([...]) require multiLine=True — without it, corrupt record error.

Quarantine Before Transform

Isolate bad records into a separate table first. Nothing gets silently dropped — always preserved for investigation.

NULL Handling in Filters

NULL < 0 evaluates to NULL (not True) in Spark. Always add isNotNull() explicitly alongside range filters.

PII in Pipelines

Raw PII never reaches the silver layer. SHA-256 via sha2() replaces identifiers while keeping join-back capability.

when().otherwise()

PySpark's CASE WHEN equivalent. Must use & / | (bitwise), not Python's and / or — those silently fail on Column objects.
